# Contributing to aptly-ui

Спасибо за интерес к проекту! Мы рады приветствовать контрибьютеров.

## Как начать разработку

### Подготовка окружения

1. Форкните репозиторий и клонируйте его локально:
```bash
git clone https://github.com/YOUR_USERNAME/aptly-ui.git
cd aptly-ui
```

2. Создайте виртуальное окружение и установите зависимости:
```bash
make install
```

3. Запустите приложение локально:
```bash
make run
```

Приложение будет доступно на `http://127.0.0.1:5000`

### Локальное тестирование

Перед отправкой pull request убедитесь, что код работает:

```bash
# Проверка синтаксиса Python
python3 -m py_compile app.py routes/*.py

# Сборка Docker образа (опционально)
make build

# Запуск контейнера (опционально)
make start
make logs
make stop
```

## Структура проекта

- **app.py** — основное Flask приложение
- **routes/** — модули с HTTP маршрутами
- **templates/** — Jinja2 HTML шаблоны
- **static/** — CSS, JavaScript
- **i18n/** — файлы локализации (en.json, ru.json)
- **docker/** — Dockerfile для контейнеризации
- **Makefile** — скрипты для разработки и деплоя

## Соглашения по коду

### Python

- Используйте PEP 8 стиль оформления кода
- Комментарии и документация могут быть на русском
- Не оставляйте `print()` вызовы — используйте логирование через `app_logging.py`
- Обрабатывайте исключения правильно

### Frontend (JavaScript)

- Находится в `static/aptly-ui.js`
- Использует ванильный JavaScript (без фреймворков)
- После изменений обновите страницу в браузере (Ctrl+F5)

### Локализация

- Все пользовательские строки используют `_t()` функцию для i18n
- Добавляйте новые строки в `i18n/en.json` и `i18n/ru.json`
- Структурируйте ключи по смыслу (например: `flash.error.message`)

## Типы контрибуций

### Баги

1. Проверьте, что баг не был уже отправлен
2. Создайте Issue с подробным описанием:
   - Как воспроизвести баг
   - Ожидаемое поведение
   - Фактическое поведение
   - Версия Python, ОС

### Новые возможности

1. Создайте Issue с описанием идеи
2. Дождитесь обсуждения
3. Создайте PR с реализацией

### Документация

- Исправления в README.md приветствуются
- Добавляйте примеры и пояснения
- Проверяйте, что примеры команд работают

## Процесс Pull Request

1. Создайте ветку от `main`:
```bash
git checkout -b feature/название-фичи
```

2. Сделайте изменения и коммиты:
```bash
git add .
git commit -m "Описание: краткое описание изменений"
```

3. Отправьте ветку:
```bash
git push origin feature/название-фичи
```

4. Создайте Pull Request на GitHub с описанием:
   - Что было изменено
   - Почему это нужно
   - Как это тестировалось
   - Связанные Issues (если есть)

## Требования к PR

- ✓ Код проходит синтаксическую проверку
- ✓ Изменения логичны и сосредоточены на одной цели
- ✓ Попутные изменения минимальны
- ✓ Комментарии и документация обновлены
- ✓ Нет trailing whitespace и неправильного форматирования
- ✓ Соблюдается PEP 8

## Лицензия

Отправляя Pull Request, вы соглашаетесь, что ваш код будет под той же лицензией, что и проект.

## Вопросы?

Создайте Issue с вопросом или обсудите в существующих Issues.
